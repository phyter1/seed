Thorough accounting of current gaps, carried-forward follow-ups from recent handoffs, and known-but-unresolved quirks. Not a priority list — that's a separate judgment call. This is the inventory.

## 1. Load-Bearing (actively accumulating cost)

### 1.1 vec0 PK disagreement — papered over with try/catch

**Where:** Every `INSERT INTO vec_memories` call site in `@seed/memory@0.4.x` is wrapped in per-row try/catch.

**Problem:** Something in `sqlite-vec`'s `vec_memories_rowids` internal table disagrees with the primary keys we think we're inserting. We catch and swallow the error, which prevents ingest from crashing but:
- The workaround replicates across every call site (call-site fan-out)
- Real errors could hide behind the same catch
- Root cause is not understood

**Next step:** Read `sqlite-vec` internals OR file an upstream issue with a minimal repro. The try/catch is tolerable for one site; it's not tolerable for N.

**Source:** `HANDOFF-memory-search-and-recall-skill-2026-04-05.md` item #5.

### 1.2 Standalone `seed.config.json` workload missing

**Where:** Fleet router artifact ships `seed.config.json` as a sidecar inside the router tarball.

**Problem:** Fleet topology (machines, providers, models) is coupled to router releases. Every topology change requires a new router build + deploy. The config should be its own workload with its own lifecycle.

**Impact:** Already bleeding — topology edits get stuck waiting for router release cadence.

**Source:** `HANDOFF-memory-search-and-recall-skill-2026-04-05.md` item #7.

---

### 1.3 Phase 1 workload installer — launchd re-bootstrap missing

**Where:** `packages/fleet/` workload installer.

**Problem:** When installing `memory@0.4.9` to ren1, the install dir was created and `~/Library/LaunchAgents/com.seed.memory.plist` was rewritten, but the existing `com.seed.memory` service was **not unloaded/reloaded**. Service was down after install until manual `launchctl bootstrap gui/$UID ~/Library/LaunchAgents/com.seed.memory.plist` was run.

**Impact:** Every memory deploy until this is fixed requires manual intervention on the target machine. Cost 10 minutes of the last session.

**Fix:** Installer should `launchctl bootout` + `launchctl bootstrap` after plist rewrite.

**Source:** `HANDOFF-memory-search-and-recall-skill-2026-04-05.md` item #9 + quirk #1.

### 1.4 Workload installer — three distinct cleanup gaps

**Audited 2026-04-05:** ren1 is carrying **~838MB of dead seed state** from accumulated workload upgrades. At current release cadence (~10 memory releases in 24h), this grows by ~74MB per release, unbounded.

**Three distinct gaps:**

**(a) Install-dir GC missing.** After `workload.install memory-0.4.9`, the previous `memory-0.4.8/` install dir isn't removed. ren1 currently holds 10 memory install dirs: `0.1.0, 0.2.0, 0.4.2, 0.4.3, 0.4.4, 0.4.5, 0.4.6, 0.4.7, 0.4.8, 0.4.9` — ~500MB. ren3 holds 2 fleet-router install dirs. Fix: installer should remove prior install dirs not matching `current + N previous` after a successful install. N should be configurable (suggest N=2 for rollback headroom).

**(b) Artifact staging never purged.** Tarballs download to `~/.local/share/seed/workload-artifacts/` (e.g. `memory-0.4.9-darwin-x64.tar.gz`), get extracted into `~/.local/share/seed/workloads/`, and then sit there forever. ren1 holds 8 stale artifact tarballs (0.4.2–0.4.9) at 24MB each = 192MB. Fix: delete the tarball after successful extraction, OR keep only the N most-recent matching the GC'd install-dir policy.

**(c) Bootstrap `/tmp` orphans.** ren1 has `/tmp/memory-0.1.0-darwin-x64.tar.gz`, `/tmp/memory-0.2.0-darwin-x64.tar.gz`, and `/tmp/seed-memory-seed.db` — dating to initial bootstrap from Apr 4. ~57MB. Never cleaned up by any automated path. Fix: early installer drafts used `/tmp` as staging; newer flow uses `workload-artifacts/`. Installer should sweep stale seed-owned `/tmp` files on startup, OR these are one-shot cleanup the operator runs once.

**Binary self-update is clean** — atomic-rename replace works correctly; zero orphan `.new.PID` or `.tmp` files across ren1/ren2/ren3. Only workload state accumulates.

**Installer launchd re-bootstrap gap** from prior handoffs (see §1.3) is a related but distinct issue.

**Source:** `HANDOFF-memory-search-and-recall-skill-2026-04-05.md` quirk, plus audit 2026-04-05.

### 1.5 `seed` CLI broken on fleet machines (config not bootstrapped)

**Where:** `packages/fleet/control/src/cli.ts`; fleet machines running the `seed` binary from `~/.local/bin/seed`.

**Problem:** On ren1 (and presumably ren3, any machine that isn't the control plane host), `seed fleet status` errors with `"Fatal: Unable to connect. Is the computer able to access the url?"` because:

1. `~/.config/seed-fleet/cli.json` is not created during base install — it's only written by `seed fleet join`, which isn't called by `install-deps.sh` or `install-control-plane.sh`.
2. `SEED_CONTROL_URL` env var is not exported to operator shells.
3. CLI falls back to `http://localhost:4310` (cli.ts:32). Control plane runs on ren2 only. Connection refused.
4. Even with the URL forced, `SEED_OPERATOR_TOKEN` is unset on ren1, so the request 401s. The token was rotated on 2026-04-05 and never propagated beyond ren2's launchd plist.

**Impact:** The CLI is non-functional from every machine except ren2 for any operator or agent workflow. Operators fall back to SSH+curl (the pre-CLI path), defeating the CLI's purpose.

**Fix surface:**
- Installer writes `~/.config/seed-fleet/cli.json` with the control URL the machine joined to.
- A secure distribution path for `SEED_OPERATOR_TOKEN` to operator machines (keychain, env file under `600`, or "show once on approve" as join already does for agent tokens).
- Better error message: include the URL the CLI tried and suggest `seed fleet join` or `SEED_CONTROL_URL=`.

### 1.6a Binary distribution principle: pull-from-GitHub, never push-via-SSH

**Architectural principle:** *Seed binaries (agent, CLI, control-plane) propagate to fleet machines exclusively via `self-update` pulling from GitHub releases. SSH/scp is not a distribution channel for Seed's product contract.*

**Already built (good):**
- `packages/fleet/control/src/self-update.ts` — platform detection, GitHub release resolution, SHA-256 verification against `checksums.txt`, atomic rename over the running binary. Shared by all three binaries.
- `cmdUpgrade` (`cli.ts:703`) — dispatches `agent.update` command over the WebSocket to each agent; each agent pulls its own binary from GitHub and restarts.
- `cmdSelfUpdate` (`cli.ts:783`) — updates the CLI binary in place.

**Still missing:**
1. **`seed fleet release --version vX.Y.Z`** — single command that (a) updates the control plane first, (b) waits for it to come back, (c) then broadcasts agent updates. Today this is two manual steps and an implicit ordering.
2. **Control-plane self-update as a distinct operation.** `seed fleet self-update` on the control-plane host currently updates the CLI binary, not `seed-control-plane`. Need `--binary control-plane` or a separate subcommand. Ideally invoked remotely from an operator machine — which means the control plane needs an endpoint that triggers its own self-update + restart.
3. **Release workflow verification.** `.github/workflows/release.yml` exists; not yet verified that tagging `v0.4.3` actually publishes all 9 binaries + `checksums.txt` that `self-update.ts` expects.
4. **Old-agent migration path.** The current fleet agents are on 0.2.x and may not implement the `agent.update` command the 0.4.x control plane would send. First upgrade from 0.2.x is a bootstrap problem — probably one-time scp is the pragmatic answer, documented as an exception.

**Impact:** The infrastructure exists. What's missing is the orchestration (single release command) and the control-plane self-update hook. Until those are filled, every release requires manual ordering by the operator.

### 1.6 Fleet skills use SSH; Seed's contract is CLI-only

**Architectural principle:** *Seed fleet operations must flow through the `seed` CLI → control plane HTTP API. SSH between fleet machines is not part of Seed's product contract.* A Seed user running the product on cloud VMs, a friend's machine, or a mixed-OS fleet cannot be assumed to have passwordless SSH mesh. The control plane is the only legitimate coordination surface.

**Where:**
- `.claude/skills/fleet-status/SKILL.md` — SSHes into each machine + curls local ports
- `.claude/skills/fleet-ssh/SKILL.md` — explicitly an SSH tool
- `.claude/skills/fleet-inference/SKILL.md` — likely SSH-based for remote model calls
- `.claude/skills/fleet-dns/SKILL.md` — Cloudflare API (fine, not fleet-internal)
- `packages/skills/` — migrated copies of the above

**Problem:** The existing fleet skills predate the control plane. They work in Ryan's setup because his 4 machines happen to have an SSH mesh. They cannot ship as part of Seed's public skill surface. They are Ryan-infra tooling that got committed to the wrong repo, or Seed-product skills written before the CLI existed.

The `seed` CLI already exposes the legitimate operations: `status`, `approve`, `revoke`, `config`, `audit`, `upgrade`, `self-update`, `installs`, `join`. **None of these are accessible to a Claude session via a skill.**

**Impact:**
- Every "check fleet" interaction in Seed today uses SSH, which won't work for anyone else.
- Install session tracking (`seed fleet installs --follow`) is unreachable from a conversation.
- `seed fleet upgrade` cannot be driven from a Claude session.
- The `fleet-status` skill will silently appear to work in Ryan's setup and silently fail for every other Seed user.

**Fix:**
1. Create `.claude/skills/seed/SKILL.md` wrapping the CLI subcommands. Allowed tools: `Bash(seed *)`. Document env-var requirements (`SEED_CONTROL_URL`, `SEED_OPERATOR_TOKEN`).
2. **Deprecate or delete the SSH-based `fleet-*` skills from the Seed repo.** If they're Ryan-personal tooling, move them to existential or a separate repo. If they're meant to be Seed skills, rewrite them against the CLI + control plane.
3. Add an explicit `CONTRIBUTING.md` or architecture note stating "skills must not assume SSH access between fleet machines."

**Blocked by 1.5** — no point wrapping a CLI that can't reach its control plane from operator machines.

**Gaps this exposes in the CLI itself** (work items for the control plane):
- No `seed fleet health <machine>` that reports service status (memory, router, MLX) on a target machine. Today the agent reports `connected` to the control plane but not what services are running on it.
- No `seed fleet models <machine>` equivalent to `curl :11434/api/tags` or `:8080/v1/models`. The control plane doesn't surface model inventory.
- No `seed fleet logs <machine> <service>` for remote log tailing.
- No `seed inference <prompt>` hitting the fleet router with the operator's configured default.

These are features the SSH-based skills provide today that the CLI would need to match before the SSH skills can be cleanly removed.

## 2. Architectural Drift

### 2.1 Split-brain filesystem contract (EPIC-001)

**Where:** README + boot file describe root-level identity files; real repo ships templates under `packages/core/identity/*.template`.

**Problem:** A new user reading the README expects `self.md`, `continuity.md`, etc. at the repo root. None exist. `packages/core/` has template scaffolding that is not rendered into root. Both layouts are half-implemented.

**Impact:** First-run behavior is undefined. Any host adapter that looks for identity files has to guess. Boot paths are non-deterministic.

**Fix:** Pick one layout (recommendation: root-level canonical, `packages/core/` becomes template source only), implement it, update all references. This is EPIC-001 and it blocks EPICs 002, 006, 009.

**Where:** `~/code/existential/heartbeat.sh` + related scripts run autonomously on ren1. `~/code/seed/packages/heartbeat/` has the host-neutral version but is **not the thing actually running in production**.

**Problem:** The running heartbeat and the seed heartbeat package have diverged. The existential version calls `claude -p` directly. The seed version uses host dispatch via env vars. Neither is being actively maintained against the other.

**Decision pending:** Does seed's heartbeat replace existential's? Or does existential stay as Ryan's personal continuity instance while seed's is the generalized product?

### 3.3 Cross-platform sqlite-vec install undocumented

**Where:** `packages/memory/scripts/build-binaries.sh`.

**Problem:** On darwin-arm64, `bun install` silently skips foreign-platform native packages. Building darwin-x64 + linux-x64 artifacts requires `npm install --no-save sqlite-vec-darwin-x64@0.1.9 sqlite-vec-linux-x64@0.1.9 --force` before running the build script. This is known and reproducible but **not in the README or the build script itself**.

**Fix:** Bake into `build-binaries.sh` OR add a "Cross-platform builds" section to the memory package README.

**Source:** `HANDOFF-memory-search-and-recall-skill-2026-04-05.md` item #6 + quirk #2.

## 4. Observability & Validation

### 4.1 No adapter smoke tests

**Where:** EPIC-010 scope.

**Problem:** `packages/hosts/src/adapters/codex.ts` and `gemini.ts` exist structurally. There is no test that verifies either actually launches its target binary with a rendered boot file. Only the Claude adapter has any implicit validation (it's the one actually used).

**Fix:** One smoke test per adapter: "render BOOT.md → spawn host binary in `--version` or equivalent mode → assert exit 0."

### 4.2 No path/doc drift validation

**Where:** No check exists that paths referenced in `README.md` actually exist in the repo.

**Impact:** The `self.md` / `continuity.md` README→reality mismatch above would have been caught at PR time.

**Fix:** Simple script in CI: extract code-spans + relative paths from README.md, assert each resolves.

### 4.3 CI only runs release workflow

**Where:** `.github/workflows/` contains only `release.yml`.

**Problem:** No PR-time test run, no lint, no typecheck. Per-package tests exist (and are comprehensive — memory has 93 tests) but they only run locally. Nothing stops a PR that breaks tests from merging.

**Fix:** Add `.github/workflows/test.yml` that runs `bun test` across packages on PRs.
